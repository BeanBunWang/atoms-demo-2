# Atoms Demo 交付说明

## 交付结论

项目已交付一个可在线运行、可公开检查源码的 Atoms-like AI App Builder：

- 在线地址：<https://atoms-demo.beanbunwang-lab.workers.dev>
- GitHub：<https://github.com/BeanBunWang/atoms-demo-2>

这不是只复刻创作页外观的静态样例。公开主页提供注册和登录；登录后用户可以不选择模式，直接输入需求。系统完成意图识别、必要澄清、动态计划、人工审批、受控工具执行、源码生成、产物校验与反馈。右侧 Reviewer 可在 Preview 与 Code 之间切换，直接修改三份源码，校验后应用为新 revision，并可回滚历史版本。项目、对话、源码、版本和交互状态会同步到 Cloudflare KV。

计算器和贪吃蛇具备领域交互运行时：计算器可以点击完成四则运算、百分比和正负号；贪吃蛇可以开始、暂停、重置、方向控制、碰撞判定和计分。同一会话继续提出修改时，系统会基于已有产物和最近对话生成新的修改计划，不复用上一轮过期意图或计划。

## 实现方案

核心闭环是：

`自然语言需求 → 意图识别 → 必要时澄清 → 动态计划 → 用户批准 → Agent/工具执行 → 结构化产物 → 校验/修复 → 可操作 App Viewer`

| 层 | 实现 | 代码位置 |
| --- | --- | --- |
| 意图与路由 | 区分构建、修改、研究、数据、增长、视频，并识别 calculator / snake 领域 | `agent/intent.mjs` |
| Agent loop | plan 与 build 两阶段运行；审批前停止；按依赖调度专家和工具；流式返回状态 | `agent/runtime.mjs` |
| 模型协议 | DeepSeek V4 Pro 输出意图、计划和紧凑应用 schema；runtime 从已验证 schema 生成可编辑源码，避免长 JSON 截断 | `agent/prompts.mjs`、`agent/model.mjs` |
| Tool use | 研究、产品、设计、应用组合、数据、增长、分镜和验证均为受控本地函数 | `agent/tools.mjs` |
| 交互产物 | `appType` 将通用 UI schema 路由到计算器、贪吃蛇或通用交互；计算器与游戏使用确定性 reducer | `site/js/interactive.js`、`site/js/app.js` |
| 增量修改 | 每轮提交清空旧计划；使用 revision、顶层 set 和稳定 ID sectionOps 生成 patch，验证通过后原子提交 | `agent/tools.mjs`、`site/js/planner.js` |
| Preview 闭环 | 验证 schema、Code 同步、行为连接和实际 DOM；失败保留上一版并可一键回灌 Agent | `site/js/preview-loop.js`、`site/js/app.js` |
| Code 与版本 | 三文件源码编辑、原子应用、Preview 校验、revision 快照和回滚 | `site/js/code-workspace.js`、`site/js/planner.js` |
| 账号与状态 | 注册、登录、HttpOnly 会话；项目、对话、源码、版本和交互状态本地缓存并同步 KV | `backend/auth.mjs`、`site/js/auth.js`、`site/js/storage.js` |
| 部署边界 | Node 服务负责本地运行；Cloudflare Worker 承担静态托管、同源 API、模型代理、KV 与限流 | `server.mjs`、`worker.mjs`、`wrangler.jsonc` |

完整预览不再导出一份容易漂移的静态 HTML，而是以 `?preview=<workspaceId>` 打开同源只读工作台视图，复用主 Viewer 的交互运行时。因此主 Viewer、完整预览和增量修改使用同一份 workspace schema。

## 技术取舍

| 选择 | 原因 | 放弃或延后的方案 |
| --- | --- | --- |
| 受控 schema + 领域 reducer | 交互可测试、可持久化，也不会执行模型注入的任意代码 | 浏览器直接执行模型生成代码；能力更自由，但需要真正的隔离沙箱 |
| 语义交给模型，执行边界确定化 | 既保留自然语言理解，又能稳定验证工具输入和产物 | 让模型直接输出命令或完整 runtime；失败面和安全风险过大 |
| plan/build 分阶段并保留审批门 | 计划可见、可改，执行边界清楚 | 收到输入立即执行；速度更快但不符合 Atoms 的可控创作体验 |
| 增量修改保存产物、不保存过期计划 | 保留应用身份、主题和内容，同时避免二次修改误用旧上下文 | 每次全量重建；实现简单但容易丢失用户已有结果 |
| local-first 缓存 + KV 云端状态 | 页面即时恢复，登录账号又能获得服务端持久化；数据模型保持简单 | 引入关系数据库与多表迁移；当前单用户工作区快照不需要该复杂度 |
| 随机盐 + 服务端 pepper HMAC + HttpOnly Cookie | 避免在浏览器保存密码或会话明文，也能满足 Cloudflare 免费层 CPU 预算 | 高迭代 PBKDF2 在免费 Worker 的 10 ms CPU 限制下不稳定；第三方 OAuth 需要额外审核 |
| Cloudflare Worker 托管与代理 | 免费层可托管静态资源和保存服务端 Secret，部署链路短 | 独立前后端与数据库；当前收益不足以抵消运维复杂度 |

## 完成边界与已知问题

- “可生成应用”当前分为两层：计算器和贪吃蛇拥有专属行为运行时；其他领域生成的是可操作 UI schema，不等于自动生成任意业务后端。
- Preview 闭环验证的是受控 schema 渲染、源码同步与领域 reducer，不是在隔离沙箱内编译执行任意模型 JavaScript。
- Code 面板允许直接修改 `App.jsx` 中受标记保护的 Preview 配置、CSS 和 JSON；会拒绝损坏标记、非法 JSON 和主动内容 CSS。它不会安装依赖或执行任意 npm 代码。
- 当前账号系统是邮箱密码登录，没有邮箱验证、找回密码、OAuth、团队成员或精细权限。
- KV 以每个账号一份完整工作区快照保存，适合 Demo；高并发协作需要版本冲突控制和数据库拆分。
- Publish 只模拟发布状态；完整预览复用当前同源工作区，不会为每个生成应用创建独立公网 URL。
- GitHub connector 是演示上下文，不会读取私人仓库；真实连接器需要 OAuth、权限最小化和审计。
- 真实模型仍可能受上游延迟或配额影响；失败时保留本地确定性计划，线上每 IP 每分钟最多 20 次 Agent API 请求。

## 后续扩展优先级

| 优先级 | 方向 | 判断 |
| --- | --- | --- |
| P0 | 隔离代码沙箱 + 编译测试修复循环 | 这是从“受控领域应用”扩展到“任意应用生成”的关键能力，必须先解决文件、网络、进程、资源和凭据隔离 |
| P1 | 独立发布、版本化和回滚 | 沙箱产物可信后，再为每个应用生成稳定 URL，支持部署记录和回滚 |
| P1 | 真实 GitHub / Figma / 数据库连接器 | 能显著扩大输入与业务能力，但需 OAuth、授权、审计和多租户数据隔离 |
| P2 | Diff、Replay 与评测集 | 对比每轮 intent、plan、tool、artifact 和验证结果，提升可解释性与回归质量 |
| P2 | 邮箱验证、找回密码与团队协作 | 基础账号和云端恢复已完成；多人协作需要角色权限、冲突合并与审计 |
