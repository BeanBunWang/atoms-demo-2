# Atoms Demo 交付说明

## 结论

项目已完成可运行的 Atoms 核心体验复刻，并部署为在线真实模型版本：

- 在线地址：<https://atoms-demo.beanbunwang-lab.workers.dev>
- GitHub：<https://github.com/BeanBunWang/atoms-demo-2>

用户无需选择模式，可以直接描述想法。系统识别意图；只有缺失信息会显著改变产品方向时，Alex 才提出结构化澄清。回答后 Mike 生成动态计划，用户批准后 Agent 执行受控工具，并在对话中持续反馈过程、产物和验证结果，最终更新 App Viewer。

App Viewer 已从静态结果展示升级为可交互的验收工作台：用户可以调整设备视口、检查代码和运行日志、在 Design 中修改内容/组件/主题，并直接验证生成应用的导航与关键操作。所有可见入口都有状态、内容、面板变化或明确反馈，关键 Viewer 状态随工作区保存。

## 实现思路与关键取舍

实现主线是把 Atoms 的创作过程收敛为一个可验证闭环：

`自然语言需求 → 意图识别 → 必要时结构化澄清 → 动态计划 → 人工批准 → Agent/工具执行 → 产物验证与修复 → App Viewer 验收`

- **两阶段 Agent runtime**：计划阶段只做意图判断、专家/工具路由和审批，不提前执行；构建阶段按依赖执行受控工具，共享结构化产物，并在结束前运行验证器。
- **真实模型与确定性边界结合**：DeepSeek Flash 负责语义理解与生成，schema 规范化、人工批准、受控工具和验证器负责稳定性，避免模型输出直接成为执行指令。
- **安全 UI schema 代替任意代码执行**：Viewer 渲染结构化应用 schema，Design、Library 和 Theme 修改同一份 schema 并同步生成代码；当前不在浏览器执行模型生成的任意代码。
- **本地优先的数据持久化**：工作区、项目列表、对话、设备、Viewer 面板、主题和预览交互状态写入 `localStorage`；支持刷新恢复和 JSON 工作区导出，不依赖账号体系。
- **同构部署边界**：Node 服务用于本地运行，Cloudflare Worker 承担线上模型代理与静态资源托管；DeepSeek Key 只存在于 `.env` 或 Cloudflare Secret。
- **对话优先的界面取舍**：模式不作为输入门槛；意图、计划、tool call/output、验证和重规划回到对话时间线，Console/Files 作为次级检查入口。

## App Viewer 完整交付

| 入口 | 行为 |
| --- | --- |
| Refresh | 从当前 workspace schema 重新渲染，重置临时选择与滚动，不丢失主题和内容修改 |
| Desktop / Tablet / Mobile | 切换预览宽度并持久化选择 |
| Design | 开关页面元素编辑状态；Hero 与具体区块使用对应内容填充编辑表单 |
| Visual Editor | 返回应用预览并支持定位、编辑生成区块 |
| Library | 展示当前结构，可添加 cards、stats、timeline、progress 受控组件并同步代码 |
| Theme | 应用四套预置主题，保存强调色、背景与标题风格 |
| Console / Files | 打开 Agent runtime 日志、专家进度和生成文件快照；抽屉不会遮挡工具栏 |
| Code | 在同一 Viewer 内切换生成代码，并保留当前工作区状态 |
| Open preview | 在新窗口打开包含全部区块、导航、CTA 和项目选择的当前快照 |
| 生成应用导航/CTA/项目/头像 | 页内定位、完成/选择状态、运行状态入口与重置操作 |

Viewer 内交互用于验证 UI schema 和产品流程，不代表已接入真实业务数据库；独立预览是当前工作区快照，Publish 仍为模拟发布。

## 如果继续投入时间：扩展方向与优先级判断

| 优先级 | 扩展方向 | 为什么按这个顺序 |
| --- | --- | --- |
| P0 | 隔离代码沙箱 | 当前最大能力边界是只能运行安全 UI schema。沙箱能把“生成代码展示”升级为安装、编译、测试、错误回灌和自动修复，是最直接的产品跃迁；前提是补齐资源、网络和凭据隔离 |
| P1 | 真实发布链路 | 用户完成构建后的下一需求是获得独立公开应用；应在沙箱产物可信后增加版本化部署、回滚和稳定链接，避免发布不可验证代码 |
| P1 | 真实连接器与后端 | GitHub、Figma、Supabase 能显著扩展输入和业务能力，但涉及 OAuth、权限、审计与数据隔离，收益高、风险也高，因此与真实发布并列第二阶段 |
| P2 | 版本 Diff 与 Replay | 有助于理解 Agent 决策、比较改动和恢复历史，但不阻断当前从需求到预览的主流程，适合在执行与发布链路稳定后补齐 |
| P2 | 公开访问防护与账号体系 | 当前已有同源校验、请求上限和每 IP 限流；当使用量或真实后端接入后，再引入 Turnstile、登录、配额和团队权限更符合投入产出比 |
