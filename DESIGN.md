# Atoms Demo 设计说明

## 产品定位

这不是一个任务看板，而是一个“AI 应用构建工作台”。用户在同一界面完成需求沟通、计划审批、构建观察、应用查看、可视编辑和发布，减少不同工具之间的上下文切换。

## 信息架构

- 左侧全局导航：项目、资源入口和本地工作区。
- 对话区：需求消息、实施计划、模式选择和后续修改指令。
- App Viewer：Preview / Code、设备视口、Design mode 和独立预览。
- 活动区：智能体状态、终端日志和文件树。

## 关键取舍

- 使用两段式 Agent runtime：第一段完成意图识别、动态路由和计划审批；第二段执行受控工具、生成产物、验证并按需重规划。
- 每个工具都输出结构化共享产物，后续专家读取上游结果，避免每个角色从零开始或只播放状态动画。
- 模型输出统一经过 schema 规范化；预览由通用 UI schema 渲染，话题既决定内容，也决定布局和模块类型。
- 预览是结构化模板，不执行任意用户代码，避免在纯静态站点中伪造不安全沙箱。
- 计划必须审批后才能构建，构建完成后才能发布，动作与系统状态保持一致。
- placeholder 由原生属性和显式 `.has-value` 状态双重约束，输入后立即隐藏。

## 视觉系统

- 白色/浅灰工具界面，蓝色只用于主操作和活动状态。
- 4px 基础间距，界面圆角 6–10px，重点容器使用细边框与轻阴影。
- 生成应用预览使用 Georgia 建立产品感，与工具界面的系统无衬线区分。
- 只使用状态过渡和消息进入动画，并尊重 `prefers-reduced-motion`。

## 能力边界

模型调用位于纯 Web API 模块 `agent/model.mjs`，由本地 `server.mjs` 与 Cloudflare `worker.mjs` 复用；`agent/intent.mjs` 负责意图，`agent/runtime.mjs` 负责生命周期，`agent/tools.mjs` 负责受控工具与验证，`agent/prompts.mjs` 负责阶段提示词；浏览器工作区位于 `site/js/`。API Key 只由服务端读取，浏览器只接收规范化事件和产物。若升级为真实产品，仍需增加隔离构建环境、真实连接器和发布提供商，不应把 API Key 或任意代码执行能力放入浏览器端。
